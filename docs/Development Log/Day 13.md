## 핵심 구현 상세

### 서비스형 프런트 앱 셸과 핵심 탐색 화면 목업 확장

#### 문제 상황
- Day 12까지는 타이머 중심 흐름만 상대적으로 구체화돼 있었고, 나머지 서비스 화면은 placeholder 수준이라 전체 제품 흐름을 한눈에 검증하기 어려웠다.
- 홈, 랭킹, 학습처럼 서비스 인상을 결정하는 핵심 탐색 화면이 비어 있어 이후 실연동 우선순위를 잡기 어려웠다.

#### 해결 방법
- `App.jsx` 기준으로 상단 네비게이션, 계정 상태 chip, footer feedback 진입점을 정리해 공통 앱 셸을 만들었다.
- `HomePage`, `RankingsPage`, `LearningPage`를 개별 페이지로 구현하고 mock 데이터 기반으로 실제 서비스처럼 탐색 가능한 상태까지 올렸다.
- `TimerPage`는 현재 스크램블, 상태 표시, 최근 기록 패널 중심의 집중형 UI로 재구성했다.

#### 선택 이유
- 먼저 서비스형 화면 세트를 완성해 두면 API 실연동 전에도 사용자 흐름과 화면 요구사항을 구조적으로 검토할 수 있다.
- 홈, 랭킹, 학습을 먼저 구체화하면 이후 백엔드 계약을 어떤 화면 상태에 맞춰 확장해야 하는지 기준이 생긴다.

#### 트레이드오프
- 목업 기반이라 화면 요구사항은 빠르게 검증할 수 있지만, 실제 API 계약과 차이가 생기면 후속 실연동 단계에서 다시 정렬해야 한다.
- 타이머를 제외한 화면은 대부분 mock 데이터 기반이라 제품처럼 보이더라도 실제 데이터 흐름이 완성된 것은 아니다.

#### 결과
- 홈, 랭킹, 학습, 타이머를 포함한 핵심 탐색 화면이 서비스형 UI로 연결됐다.
- 이후 일정에서 어떤 화면부터 실연동하고 어떤 API를 확장해야 하는지 더 명확해졌다.

#### 한 줄 요약
- 앱 셸과 핵심 탐색 화면을 먼저 완성해 서비스 전체 흐름을 눈으로 검증할 수 있는 기준선을 만들었다.

#### 문제 상황

## 트러블슈팅 / 문제

- 문제: `/timer`에서 `Space`를 누르면 타이머 상태 전이와 함께 페이지가 아래로 스크롤되는 현상이 있었다.
- 원인: 일부 상태에서 브라우저 기본 `Space` 스크롤 동작이 키 이벤트 처리보다 먼저 살아남을 수 있었고, 이벤트를 더 이른 단계에서 소비하지 못하고 있었다.
- 해결: `keydown`, `keyup`, `keypress`를 더 이른 단계에서 차단하고 `preventDefault()`를 앞쪽으로 이동시키는 방향으로 정리한 뒤, 재발 방지 포인트를 별도 트러블슈팅 문서로 기록했다.
- 남은 리스크: 이후 다른 페이지에 전역 keyboard shortcut이 늘어나면 타이머 화면의 `Space` 처리와 충돌하지 않는지 다시 확인해야 한다.
