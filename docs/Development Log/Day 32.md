# Day 32 - CFOP 공식 데이터 중복 보정

> 이 로그는 `2026-04-28`에 진행한 학습 화면 CFOP 정적 데이터 중복 보정과 frontend coverage 100% 유지 작업을 정리한 기록이다.

## 작업 배경

`Day 31` 초보자 학습 탭 작업에서 기존 CFOP 데이터의 중복 공식은 수정하지 않고 후속 TODO로 남겼다. 이후 학습 화면 리디자인이 마무리된 상태에서 `F2L`, `OLL`, `PLL` 정적 데이터의 신뢰도를 다시 확인했다.

## 변경 내용

- `frontend/src/constants/mockLearning.js`의 CFOP 정적 데이터에서 중복으로 보이던 5묶음을 보정했다.
  - `F2L 03/F2L 30`
  - `F2L 07/F2L 26`
  - `OLL 24/OLL 28`
  - `OLL 44/OLL 45`
  - `OLL 46/OLL 47`
- 저장소 안에 생성 스크립트가 없어서 `Auto-generated by script` 주석을 현재 관리 방식에 맞는 정적 데이터 주석으로 바꿨다.
- `frontend/src/constants/mockLearning.test.js`를 추가해 CFOP 데이터 품질을 테스트로 고정했다.
  - 탭의 `itemCount`와 실제 케이스 수가 일치하는지 확인한다.
  - `F2L`, `OLL`, `PLL` stage 안에서 같은 `algorithm` 문자열이 반복되지 않는지 확인한다.
  - 같은 stage 안에서 VisualCube URL이 반복되지 않는지 확인한다.
- coverage 100% 기준을 유지하기 위해 기존 테스트의 남은 branch 사각지대를 보강했다.
  - `RankingsPage.test.jsx`: 포디움 닉네임 fallback, 내 순위 행 강조, 내 순위 empty 카드 분기를 추가했다.
  - `MyPage.test.jsx`: 빈 닉네임 프로필 이니셜 fallback 분기를 추가했다.

## 남은 리스크

- CFOP 대표 공식은 출처와 큐버 선호에 따라 달라질 수 있다. 이번 작업은 중복으로 보이던 5묶음을 `SpeedCubeDB` 기준으로 보정한 것이며, CFOP 119개 전체를 새 출처 기준으로 전면 감사한 작업은 아니다.
- VisualCube는 공식 문자열을 기반으로 그림을 만든다. 자동 테스트는 URL 중복을 막지만, 모든 케이스 그림이 실제 사용자 눈에 기대한 방향으로 보이는지까지 완전 보장하지는 않는다.
- 기존 Node 경고 `--localstorage-file was provided without a valid path`는 계속 출력된다. 테스트와 build 실패로 이어지지는 않았다.

## 후속 작업

- CFOP 119개 전체를 더 엄격히 관리하려면 기준 출처, 번호 체계, 대표 공식 선택 기준을 먼저 문서화한 뒤 전체 감사를 별도 작업으로 진행한다.
- 학습 화면에 자체 다이어그램을 추가하려면 저작권 문제가 없는 자체 상태 표현 방식을 별도로 설계한다.
