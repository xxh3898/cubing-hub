# 타이머 Space 스크롤 트러블슈팅

## 문제

`frontend/src/pages/TimerPage.jsx`에서 타이머를 사용할 때 `Space`를 누르면 타이머 상태 전이와 함께 브라우저 화면이 아래로 스크롤되는 현상이 있었다.

이 문제는 타이머 UX에 직접적인 영향을 끼친다.

- 사용자가 스페이스바로 홀드/시작/정지를 제어해야 하는데 화면 위치가 흔들림.
- 타이머 집중형 UI가 무너짐.
- 특히 긴 페이지나 모바일 뷰에서 체감이 더 클 것으로 예상.

## 원인

초기 구현에서는 `useCubeTimer` 훅에서 `keydown`, `keyup` 이벤트 안에서 일부 상태에만 `event.preventDefault()`를 호출하고 있었다.

이 방식의 문제는 아래와 같았다.

- 타이머 비활성 상태나 특정 전이 전 상태에서는 브라우저 기본 `Space` 스크롤 동작이 먼저 살아남을 수 있다.
- 이벤트를 일반 버블 단계에서만 받고 있어서 브라우저 기본 동작보다 충분히 빠르게 막지 못하는 경우가 생길 수 있다.
- `keydown`만으로는 환경에 따라 기본 스크롤을 완전히 차단하지 못하는 케이스가 남을 수 있다.

## 수정 내용

수정 파일:

- `frontend/src/hooks/useCubeTimer.js`

적용한 대응은 아래와 같다.

1. `Space` 입력이면 타이머 활성 여부를 보기 전에 먼저 `event.preventDefault()`를 시도하도록 변경
2. `keydown`, `keyup`를 캡처 단계에서 등록해서 기본 스크롤보다 먼저 이벤트를 소비하도록 변경
3. `keypress`도 추가로 막아서 브라우저별 차이를 줄임
4. `input`, `textarea`, `button`, `select` 같은 인터랙티브 요소는 기존처럼 예외 처리 유지

핵심 방향은 "타이머 화면에서 스페이스는 스크롤 키가 아니라 타이머 제어 키"라는 점을 명확히 강제하는 것이다.

## 확인 결과

수정 후에는 아래가 기대 동작이다.

- 일반 화면 영역에서 `Space`를 눌러도 페이지가 스크롤되지 않는다.
- 타이머는 기존 상태 전이대로 동작한다.
	- `idle -> holding -> ready -> running -> stopped`
- 폼 요소에 포커스가 있는 경우에는 기존 입력 동작을 방해하지 않는다.

## 재확인 방법

1. `frontend`에서 앱 실행
2. `/timer` 진입
3. 화면 아무 곳이나 클릭해 타이머 페이지에 포커스가 있는 상태 만들기
4. `Space`를 길게 누르고 떼기
5. 아래 항목 확인
   - 페이지가 아래로 스크롤되지 않는가
   - 타이머 상태가 정상 전이되는가
   - 다시 `Space`를 눌렀을 때 정지가 되는가

## 재발 시 점검 포인트

- 전역 `keydown`/`keyup` 리스너가 다른 곳에서 중복 등록되어 있지 않은지 확인
- 타이머 외 페이지에서 `Space`를 별도 단축키로 사용하고 있지 않은지 확인
- `preventDefault()` 호출 위치가 상태 분기 아래로 밀려나지 않았는지 확인
- 이벤트 등록 옵션에서 `capture: true`가 빠지지 않았는지 확인
