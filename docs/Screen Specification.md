# Screen Specification

## 1. 페이지 정보

- 문서 범위: `frontend/src/App.jsx` 기준 사용자 화면 전체
- 대상 플랫폼: PC / 모바일 웹
- 현재 사용자 화면 수: 11개
- 보조 라우트: 인증 리다이렉트 1개 (`/auth`)
- 기준 원칙:
  - 실제 API 연동 여부를 화면별로 명시한다.
  - mock 기반 화면과 실연동 화면을 구분한다.
  - 공통 상태 정책은 상단에 정의하고, 화면별로 필요한 차이만 추가한다.

## 3. 사용자 시나리오

### 시나리오 1. 비로그인 사용자의 탐색

1. 사용자는 홈에서 오늘의 스크램블과 서비스 구조를 확인한다.
2. 타이머 화면으로 이동해 실제 타이머 UX를 체험한다.
3. 로그인 전에는 기록 자동 저장 대신 안내 메시지를 본다.

### 시나리오 2. 로그인 사용자의 기록 저장

1. 사용자는 로그인 후 타이머 화면에 진입한다.
2. 종목별 스크램블을 불러오고, 스페이스바 홀드 후 측정을 시작한다.
3. 기록 정지 후 `POST /api/records`가 호출되고 최근 저장 기록 목록이 갱신된다.

### 시나리오 3. 랭킹 및 커뮤니티 탐색

1. 사용자는 랭킹 화면에서 종목과 닉네임으로 결과를 좁힌다.
2. 커뮤니티 목록에서 검색/필터 후 상세 화면으로 이동한다.
3. 현재 프런트는 mock 데이터를 사용하지만, 최종적으로는 실제 API와 연결될 예정이다.

### 시나리오 4. 회원가입 후 개인화

1. 사용자는 회원가입에서 이메일, 닉네임, 주 종목을 입력한다.
2. 로그인 후 마이페이지에서 프로필과 기록 요약을 본다.
3. 현재는 mock 상태지만, 최종적으로는 인증 연동과 대시보드 API가 연결된다.

## 4. 공통 상태 및 예외 정책

| 상태 | 의미 | 현재 적용 화면 | 처리 원칙 |
| --- | --- | --- | --- |
| `loading` | API 호출 또는 초기 데이터 준비 중 | 타이머(스크램블), 추후 홈/랭킹/커뮤니티/마이페이지 | 사용자 입력을 막아야 하는 구간만 제한하고, 진행 중 메시지를 노출한다. |
| `empty` | 데이터는 정상 조회됐지만 표시할 항목이 없음 | 타이머 최근 기록, 랭킹 mock 결과 0건 시, 추후 커뮤니티/마이페이지 | 오류처럼 보이지 않도록 안내 메시지로 처리한다. |
| `error` | API 실패 또는 유효하지 않은 요청 | 타이머 스크램블/기록 저장, 추후 전 화면 | 재시도 가능 여부를 함께 보여준다. |
| `auth failure` | 인증 필요 또는 만료 | 현재 타이머 비로그인 저장 시 안내, 추후 보호 화면 | 로그인 유도 또는 재로그인 흐름으로 연결한다. |

- 상태 및 예외
  - validation error
  - 제출 성공/실패 메시지
- 사용자 액션
  - 피드백 종류 선택
  - 제목/본문 입력
  - 제출
- 현재 상태
  - 현재는 로컬 폼 처리만 구현되어 있다.
  - 관리자 메일 전달 및 백엔드 저장/전송 로직은 구현 예정이다.

### 인증 리다이렉트

- 레이아웃 구조
  - 별도 화면 없이 라우팅 리다이렉트
- 주요 UI 요소
  - 없음
- 화면 데이터 요구사항
  - `/login` 경로 리다이렉트
- 상태 및 예외
  - 인증 완료 후 복귀 경로 설계는 추후 필요할 수 있다.
- 사용자 액션
  - 없음
- 현재 상태
  - `/auth`는 `/login`으로 이동하도록 구현되어 있다.

## 7. UX / 화면 설계 판단

### 설계 선택 1

- 선택한 방식:
  - 타이머 시작을 `홀드 300ms → Ready → 손을 떼면 시작` 흐름으로 설계했다.
- 선택 이유:
  - 큐빙 타이머 UX에서는 실수 방지와 준비 상태 인지가 중요하다.
  - 즉시 시작 구조보다 실제 큐빙 타이머에 가까운 상호작용을 만들 수 있다.
- 검토한 대안:
  - 스페이스바를 누르는 즉시 시작하는 단순 타이머
- 대안을 배제한 이유:
  - 오동작이 많고 큐빙 타이머 특유의 준비 동작을 반영하지 못한다.
- 트레이드오프:
  - 구현이 복잡해지고 상태 관리가 늘어난다.
- 기대 효과:
  - 사용자 경험이 종목 특성과 더 잘 맞고, 기록 저장 흐름과 자연스럽게 연결된다.

### 설계 선택 2

- 선택한 방식:
  - 실연동 우선 화면을 `타이머`로 두고 나머지 화면은 mock 기반으로 단계적 전환한다.
- 선택 이유:
  - 타이머는 이 서비스의 핵심 가치와 백엔드 연동 가치를 동시에 보여준다.
  - 우선 핵심 슬라이스를 닫은 뒤 다른 화면을 순차 연동하는 편이 현실적이다.
- 검토한 대안:
  - 전 화면을 한 번에 실제 API로 교체
- 대안을 배제한 이유:
  - 미구현 API가 남아 있는 상태에서 프런트 전체를 동시에 전환하면 실패 지점이 많아진다.
- 트레이드오프:
  - 현재는 일부 화면이 mock 상태로 남아 있어 완성도가 균일하지 않다.
- 기대 효과:
  - 핵심 기능부터 실제 동작을 보여주면서도 전체 제품 구조를 유지할 수 있다.

### 설계 선택 3

- 선택한 방식:
  - 공통 상태 정책(`loading`, `empty`, `error`, `auth failure`)을 화면 전체 기준으로 먼저 정의한다.
- 선택 이유:
  - mock 화면이 실제 API와 연결될 때 가장 자주 놓치는 부분이 예외 상태 설계다.
  - 화면별 설계 이전에 공통 원칙을 세워야 연동 시 품질이 일정해진다.
- 검토한 대안:
  - 실제 API 연동 시점에 화면별로 개별 대응
- 대안을 배제한 이유:
  - 상태 처리 방식이 화면마다 달라져 UX 일관성이 깨질 수 있다.
- 트레이드오프:
  - 현재 mock 단계에서 다소 과하게 보일 수 있다.
- 기대 효과:
  - 추후 실연동 전환 시 필요한 예외 처리를 미리 문서로 고정할 수 있다.
